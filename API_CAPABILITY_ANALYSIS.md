# 飞书云文档API能力梳理与分析

## 一、飞书云文档OpenAPI能力范围

### 1.1 可直接调用的API功能

#### 云空间与文件夹管理
- 获取我的空间（根文件夹）元数据
- 获取文件夹中的文件清单
- 获取文件夹元数据
- 新建文件夹
- 移动/删除文件夹（支持异步任务）
- 查询异步任务状态

#### 文件管理
- 获取文件元数据（批量查询）
- 获取文件统计信息（阅读人数、阅读次数、点赞数）
- 获取文件访问记录
- 创建云文档（文档、电子表格、多维表格）
- 复制文件
- 移动文件/文件夹
- 删除文件/文件夹
- 创建文件快捷方式
- 搜索云文档

#### 文件导入导出
- 导入文件（Word、TXT、Markdown、Excel → 云文档）
- 导出云文档（文档→Word/PDF、表格→Excel/CSV）
- 创建导入/导出任务
- 查询任务结果
- 下载导出文件

#### 素材管理
- 上传素材（图片、视频，支持分片上传）
- 下载素材（支持分片下载）
- 获取素材临时下载链接（24小时有效期）

#### 版本管理
- 创建文档版本
- 获取文档版本列表
- 获取文档版本信息
- 删除文档版本

#### 事件订阅
- 订阅云文档事件（编辑、标题变更、已读、协作者变更等）
- 查询订阅状态
- 取消订阅

### 1.2 需要本地封装的业务逻辑

#### 1.2.1 文档内容操作
**飞书API现状：**
- 仅支持通过[云文档内容API](https://open.feishu.cn/document/server-docs/docs/docx-v1/overview)进行块级读写
- Docx格式支持更丰富的内容结构

**需要本地封装的原因：**
1. **块级编辑需要精细化管理**：飞书API提供的是块级操作（插入、替换、删除块），需要本地封装成更友好的接口
2. **实时协作冲突处理**：多个用户同时编辑时需要本地实现冲突检测与合并策略
3. **内容预览与渲染**：需要本地将飞书Docx/XML格式转换为可预览的HTML/Markdown

**技术实现方案：**
```
本地服务封装层：
├── 内容缓存层（Redis缓存高频访问的块内容）
├── 冲突检测模块（基于版本号的乐观锁）
├── 格式转换模块（Docx ↔ Markdown ↔ HTML）
└── 批量操作封装（将多个块操作合并为一个请求）
```

#### 1.2.2 复杂权限组合
**飞书API现状：**
- 基础权限：查看、评论、编辑、管理
- 支持协作者添加/移除

**需要本地封装的原因：**
1. **细粒度权限控制**：飞书仅提供4级基础权限，业务可能需要更细粒度（如：仅编辑特定章节）
2. **权限继承与组合**：文件夹权限需要递归继承到子文件
3. **临时权限**：支持有效期权限、IP限制等高级特性

**技术实现方案：**
```
本地权限系统：
├── 权限表（MySQL存储自定义权限规则）
├── 权限缓存（Redis缓存用户-文档权限映射）
├── 权限继承计算器
└── 权限验证中间件
```

#### 1.2.3 历史版本对比
**飞书API现状：**
- 支持版本列表查询
- 支持获取单个版本信息
- 但**不支持版本内容对比**

**需要本地封装的原因：**
1. **版本内容差异展示**：用户需要直观看到两个版本的差异
2. **差异高亮与回滚**：支持选择性回滚部分内容
3. **版本对比性能优化**：避免每次都从飞书拉取两个完整版本

**技术实现方案：**
```
版本管理模块：
├── 版本内容快照存储（MySQL内容片段表）
├── 差异算法（基于diff-match-patch）
├── 差异可视化生成器
└── 选择性回滚功能
```

#### 1.2.4 内容索引与搜索优化
**飞书API现状：**
- 支持基础关键词搜索
- 搜索范围受限（仅用户可见文档）

**需要本地封装的原因：**
1. **全文索引**：本地建立Elasticsearch/MySQL全文索引，支持复杂搜索
2. **标签分类**：支持自定义标签体系
3. **搜索统计与推荐**：基于用户行为的搜索优化

**技术实现方案：**
```
搜索与索引模块：
├── 内容片段同步（监听飞书事件，同步到本地）
├── 全文索引构建
├── 标签管理
└── 搜索结果聚合与排序
```

## 二、技术实现理由

### 2.1 为什么选择Node.js作为服务端

| 特性 | 优势 |
|------|------|
| **异步I/O** | 飞书API调用、数据库操作、Redis操作都是I/O密集型，Node.js的事件循环模型非常适合 |
| **生态丰富** | `@larksuiteoapi/node-sdk`官方SDK支持完善，Express/Koa等框架成熟 |
| **开发效率** | JavaScript/TypeScript语言灵活，前后端技术栈统一 |
| **水平扩展** | 无状态设计，配合负载均衡容易水平扩展 |

### 2.2 为什么使用Redis缓存Token

| 问题 | Redis解决方案 |
|------|--------------|
| **token有效期** | 飞书token有效期2小时，Redis设置过期时间自动刷新 |
| **限流风险** | 避免重复请求token接口（飞书有限流） |
| **多实例共享** | 分布式部署时多个Node.js实例共享同一个token |
| **性能优化** | 内存级读取，减少网络请求 |

### 2.3 为什么MySQL存储元数据，Redis缓存热点数据

| 数据类型 | 存储方案 | 理由 |
|----------|----------|------|
| 文档元数据、权限配置 | MySQL | 结构化数据，需要事务支持，永久存储 |
| 高频访问文档元数据 | Redis缓存 | 减少数据库压力，提升响应速度 |
| 分布式锁 | Redis | 防止并发操作冲突（如：同时编辑同一文档） |
| 临时任务状态 | Redis | 导入/导出任务状态，自动过期清理 |

## 三、API能力映射表

| 业务功能 | 飞书API | 本地封装 | 说明 |
|----------|---------|----------|------|
| 创建文档 | ✅ 直接调用 | ❌ 不需要 | `POST /open-apis/drive/v1/files/create` |
| 读取文档内容 | ✅ 直接调用 | ✅ 需要 | Docx块操作，本地封装成友好接口 |
| 编辑文档 | ✅ 直接调用 | ✅ 需要 | 块级编辑 + 冲突处理 |
| 权限设置 | ✅ 直接调用 | ✅ 需要 | 基础权限用飞书，复杂权限本地 |
| 版本查询 | ✅ 直接调用 | ❌ 不需要 | `GET /open-apis/drive/v1/files/:file_token/versions` |
| 版本对比 | ❌ 不支持 | ✅ 必须 | 本地实现diff算法 |
| 文件夹管理 | ✅ 直接调用 | ❌ 不需要 | 完整支持 |
| 搜索文档 | ✅ 基础支持 | ✅ 增强 | 本地全文索引增强搜索能力 |
